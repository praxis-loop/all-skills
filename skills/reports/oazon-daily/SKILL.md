# Oazon 每日工作日报生成

## 流程

### 1. 获取当前日期和已有内容

- 读取 `daily.md` 文件（路径：D:\obsidian\xan\WorkLog\Oazon\daily.md）
- 检查今天日期是否已存在条目
- 如果已存在，读取已有内容，并检查所有工作项是否都包含时长（格式：`(Xh)` 或 `(X.Xh)`）
  - 如果有工作项缺少时长，**跳过步骤2，直接进入【补充时长模式】**
  - 如果所有工作项均已有时长，告知用户今天已有完整记录并退出
- 日期格式与文件保持一致（如 `3.13`，即 月.日，不含年份前缀，2026年的条目在 `## 2026` 下方）

### 补充时长模式

当今天的条目已有内容但部分或全部缺少时长时，进入此模式（无需询问用户）：

1. 展示当前已有内容，标出哪些条目已有时长、哪些缺失
2. 统计已有时长之和，计算剩余需要分配的时长（待分配 = 8h - 已有）
3. 为缺少时长的条目按复杂度合理分配：
   - 排查/修复类：1-2h
   - 开发/设计类：1.5-3h
   - 服务器维护类：0.5-1h
   - 文档/整理类：0.5-1.5h
4. **保留已有时长不变，只填充缺少时长的条目**
5. 逐项相加验算总和 = 8h，向用户展示结果，确认后写入文件（原地修改今天的条目，不新增条目）

### 2. 收集用户今日工作内容

如果 $ARGUMENTS 不为空，且内容是工作事务描述（而非元指令如"更新skill"），直接使用其中的内容作为今日事务。

如果 $ARGUMENTS 为空，使用 AskUserQuestion 询问以下问题：

**问题1**："今天主要做了哪些工作内容？（列出关键事项即可，AI会帮你润色扩展）"
- 自由文本输入

**问题2**："是否需要将部分内容拆分到明天？"
- 选项：不需要（今天凑满8h） / 需要拆分（指定今天保留的小时数）

### 3. AI润色与时长分配规则

**核心规则：总工时必须精确等于 8h，写入前必须逐项相加验算！**

- 每项工作时长以 0.5h 为最小单位
- 根据工作复杂度合理分配时长参考：
  - 会议类：按实际时长
  - 排查/修复类：1-2h
  - 开发/设计类：1.5-3h
  - 学习/调研类：1-2h
  - 联调/测试类：1-2h
  - 文档/整理类：0.5-1.5h
- 润色要求：
  - 保持技术性描述，简洁专业
  - 不要过度修饰，符合工程师日报风格
  - 参考 daily.md 中最近条目的写作风格和分类方式

### 4. 拆分到多天的处理

如果用户选择拆分：
- 询问今天保留多少小时（剩余分配到明天）
- **每一天各自必须精确等于 8h**
- 明天的内容应体现为"深化/继续/完善/实践"等延续性描述
- 明天也需要凑满 8h，不足的部分基于同一主题合理扩展

### 5. 写入格式

严格遵循 daily.md 已有格式，标准结构：

```
X.XX
- 项目/分类名
	- 具体工作内容(Xh)
	- 具体工作内容(Xh)
- 项目/分类名
	- 具体工作内容(Xh)
```

格式要求：
- 使用 tab 缩进（不是空格）
- 时长格式为 `(Xh)` 或 `(X.Xh)`
- 项目分类名不带时长
- 新条目插入到已有最后一个日期条目之后、待办事项（`- [x]` / `- [ ]`）之前
- 日期之间空一行

### 6. 写入前验证（必须执行）

1. 逐项列出时长并相加，确认总和 = 8h
2. 如有拆分，每天分别验算 = 8h
3. 向用户展示最终内容及验算结果
4. 用户确认后再写入文件
